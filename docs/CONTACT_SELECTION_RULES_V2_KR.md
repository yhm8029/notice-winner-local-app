# 연락처 선택 규칙 v2 설계

## 1. 배경

최근 `화담산 치유센터 건립공사 건축설계공모` 케이스에서 실제 공고문 연락처인 `건축주택과 / 053)665-2975` 대신, 지침서에 있는 `공원녹지과 / 053-665-4304`가 선택됐다.

직전 핫픽스에서는 아래 두 가지를 막았다.
- 괄호형 전화번호(`053)665-2975`) 인식 실패 수정
- 공고문에서 추출된 연락처가 있으면 지침서가 무조건 덮어쓰지 못하게 수정

하지만 이건 회귀 차단용 핫픽스일 뿐, 최종 규칙은 아니다.
공고문의 연락처도 잘못되거나 덜 적합할 수 있고, 지침서/과업지시서가 더 정확한 실무 담당을 담는 경우도 있다.

연락처 선택 규칙 v2의 목적은 `문서 종류 단일 우선순위`가 아니라 `후보 점수화 + 충돌 검토`로 바꾸는 것이다.

## 2. 목표

1. 그럴싸한 오추출을 줄인다.
2. 연락처 선택 근거를 설명 가능하게 만든다.
3. 문서 종류, 추출 위치, 라벨 강도를 함께 반영한다.
4. 충돌 시 억지 확정보다 검토 대상으로 보내는 경로를 만든다.

## 3. 비목표

1. 사람 이름까지 정확하게 구조화하는 것
2. 전화번호 포맷 통일 이상의 번호 검증
3. LLM을 1차 선택 엔진으로 사용하는 것

## 4. 기본 원칙

### 원칙 1. 문서 종류만으로 확정하지 않는다
- `공고문`이라고 무조건 우선하지 않는다.
- `지침서`라고 무조건 열위로 두지도 않는다.
- 문서 종류는 하나의 점수 요소일 뿐이다.

### 원칙 2. 추출 위치가 문서 종류보다 중요할 수 있다
- `문의처`, `연락처`, `담당부서`, `전화` 같은 명시 라벨 근처에서 추출된 후보는 강하게 본다.
- 본문 서술 중 우연히 잡힌 부서/전화는 약하게 본다.

### 원칙 3. 같은 후보가 여러 문서에서 반복되면 강하게 본다
- 동일 부서/전화 조합이 둘 이상의 문서에서 반복되면 점수를 가산한다.

### 원칙 4. 점수 차가 작으면 확정하지 않는다
- 최고 점수 후보와 차점 후보의 차이가 작으면 `CONTACT_CONFLICT` 또는 `NATIVE_REVIEW_REQUIRED`로 보낸다.

## 5. 후보 입력 단위

각 후보는 최소 아래 필드를 가진다.
- `contact_text`: 정규화된 최종 후보 문자열 (`부서/전화`)
- `raw_contact_text`: 원문 후보 문자열
- `document_type`: `announcement`, `result_announcement`, `guideline`, `task`, `other`
- `document_name`
- `document_index`
- `context_text`: 추출된 주변 문맥
- `matched_phone`
- `matched_department`
- `cue_type`: `explicit_label`, `phone_context`, `fallback_line`, `llm`
- `line_distance_to_cue`
- `confidence_flags`: 명시 라벨, 다중 문서 반복, 기관명 일치 등

## 6. 문서 종류 분류

### 6.1 우선 분류
- `announcement`
  - 공고문
  - 입찰공고문
  - 설계공모 공고문
- `result_announcement`
  - 결과공고
  - 당선작 발표 공고
- `guideline`
  - 설계공모지침서
  - 지침서
- `task`
  - 과업지시서
  - 과업내용서
- `other`
  - 참고자료, 별첨, 제안서 양식 등

### 6.2 분류 실패 시
- 문서명만으로 분류가 안 되면 `other`
- `other`는 낮은 기본점수로 시작

## 7. 점수 모델

총점은 100점 만점이 아니라 가산/감산형 상대 점수로 본다.
최종 비교만 안정적으로 되면 된다.

### 7.1 기본 문서 점수
- `announcement`: +30
- `result_announcement`: +26
- `guideline`: +18
- `task`: +14
- `other`: +8

### 7.2 추출 위치/라벨 점수
- `문의처`, `연락처`, `담당부서`, `전화`, `☎`, `☏` 같은 명시 라벨 바로 옆: +30
- 같은 줄 또는 인접 1줄 내 전화-부서 결합: +22
- 인접 2줄 이내 전화 문맥: +14
- 본문 fallback line: +6
- LLM 후보: +4 (단, 단독 확정 불가)

### 7.3 문맥 품질 점수
- `matched_department` 존재: +10
- `matched_phone` 존재: +10
- 부서명에 `과`, `팀`, `센터`, `사업소`, `담당관`, `본부` 포함: +6
- 기관명/발주처명 일부가 함께 등장: +6
- 같은 후보가 다른 문서에서도 반복: 문서 1개 추가마다 +8

### 7.4 감점 규칙
- 사람 이름/직함 비중이 높고 부서명이 약함: -8
- `문의`, `전화` 라벨 없이 본문 중간에서 우연히 잡힘: -6
- 계약/입찰 일반 문구 근처에서만 등장: -6
- 문서 종류와 문맥이 충돌 (`other` + 약한 문맥): -8
- 부서는 자연스럽지만 번호 형식이 불안정: -5

## 8. 확정 규칙

### 8.1 자동 확정
아래를 모두 만족하면 자동 확정한다.
- 최고 점수 후보가 존재
- 최고 점수 >= 45
- 최고 점수와 차점 점수 차 >= 12

### 8.2 보수 확정
아래를 만족하면 자동 확정하되 provenance에 `medium_confidence` 표시
- 최고 점수 >= 38
- 최고 점수와 차점 점수 차 >= 8
- 최고 후보가 명시 라벨 근처에서 추출됨

### 8.3 검토 필요
아래 중 하나면 검토 대상으로 둔다.
- 최고 점수 < 38
- 최고/차점 차이 < 8
- 공고문 후보와 지침서 후보가 서로 다르고 둘 다 35점 이상
- 후보가 3개 이상이고 상위권이 비슷함

이 경우 출력 정책
- `demand_contact`는 비우거나 보수 후보만 넣고
- `demand_contact_source`는 `NATIVE_REVIEW_REQUIRED` 또는 `CONTACT_CONFLICT`
- provenance에 모든 상위 후보를 남긴다.

## 9. 선택 규칙표

| 상황 | 선택 규칙 |
|---|---|
| 공고문 명시 라벨 블록에서 추출, 지침서는 약한 본문 후보 | 공고문 후보 선택 |
| 공고문 후보가 약한 본문 fallback, 지침서가 명시 라벨 블록 | 지침서 후보 역전 허용 |
| 공고문/지침서 모두 명시 라벨 블록, 후보 불일치 | 자동 확정 금지, conflict |
| 동일 후보가 공고문 + 지침서에 반복 | 동일 후보 확정 |
| 지침서/과업지시서만 후보 존재, 점수 충분 | 해당 후보 선택 |
| LLM 후보만 존재 | 단독 확정 금지 |

## 10. provenance 저장 제안

최종 필드와 함께 아래를 저장한다.
- `demand_contact_source`
- `demand_contact_confidence` (`high`, `medium`, `low`, `conflict`)
- `demand_contact_document_type`
- `demand_contact_document_name`
- `demand_contact_evidence_excerpt`
- `demand_contact_alternative_candidates` (상위 N개)

최소 1차 구현에서는 전부 DB 컬럼으로 늘리지 않아도 된다.
우선은 row meta 또는 audit/debug payload로 시작해도 된다.

## 11. 구현 단계 제안

### 1차
- 기존 `extract_contact_from_notice_text()` 후보를 단일 문자열이 아니라 후보 리스트로 반환 가능한 내부 함수로 확장
- attachment별 후보를 모아 점수화
- 현재 `_select_best_attachment_contact()`를 점수 기반 선택기로 교체
- 차점과 점수 차를 함께 계산

### 2차
- `CONTACT_CONFLICT` / `NATIVE_REVIEW_REQUIRED` 경로 추가
- provenance/debug 필드 저장
- tracker/다운로드에서 근거 확인 가능하게 연결

### 3차
- low confidence 케이스에만 LLM 보조 후보 사용
- 단, LLM은 후보 보정/설명에만 사용하고 최종 확정은 규칙이 담당

## 12. 테스트 우선순위

필수 회귀 테스트
1. `053)665-2975` 같은 괄호형 번호 인식
2. 공고문 명시 문의처 vs 지침서 다른 연락처 충돌
3. 공고문 본문 약한 후보 vs 지침서 명시 라벨 후보 역전
4. 동일 후보가 여러 문서에서 반복될 때 반복 가점
5. 대구/광주 같은 지역명과 무관하게 연락처 선택이 흔들리지 않는지

## 13. 이번 설계의 의도

핫픽스의 핵심은 `명백한 회귀 방지`였다.
연락처 선택 규칙 v2의 핵심은 `문서 종류 우선순위`를 넘어 `점수 기반 선택 + 충돌 보류`로 옮기는 것이다.

즉 최종 목표는:
- 공고문이면 무조건 우선
- 지침서면 무조건 후순위
가 아니라,
- **더 설명 가능한 후보를 선택하고, 애매하면 자동 확정하지 않는다**
이다.

## 14. 리뷰 요청 포인트

1. `자동 확정 임계값(45/38)`이 과한지
2. `차점과의 점수 차(12/8)` 기준이 적절한지
3. `result_announcement` 기본 점수를 `announcement`보다 낮게 둘지 같은 수준으로 둘지
4. `CONTACT_CONFLICT`일 때 최종 필드를 비울지, 보수 후보를 남길지
5. provenance 저장을 1차부터 컬럼으로 갈지, meta/debug로 시작할지
6. LLM은 3차로 미루는 게 맞는지
