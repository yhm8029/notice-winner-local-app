# 로그인 이력 및 관리자 로그 카드 레이아웃 설계

작성일: 2026-04-01

## 목적

운영자 모드에서만 볼 수 있는 감사 정보 영역을 확장한다.

- 성공한 로그인만 별도 이력으로 남긴다.
- 관리자 화면 카드를 `2열 x 3행`으로 재배치한다.
- `조직 감사 로그`, `다운로드 이력`, `로그인 이력`은 기본 5개만 노출하고 `더 보기`로 5개씩 확장한다.

이번 작업은 운영 중인 핵심 데이터 구조를 바꾸지 않고, 보조 감사 테이블 1개와 관리자 화면 확장만 추가하는 것을 목표로 한다.

## 범위

포함:

- 로그인 성공 감사 로그 저장
- 관리자 전용 로그인 이력 조회 API
- 관리자 화면 `2 x 3` 카드 레이아웃
- 로그 카드 3종의 `기본 5개 + 더 보기`

제외:

- 실패 로그인 시도 기록
- 일반 사용자 UI 행동 로그
- 파일 자체 저장 또는 외부 유출 차단
- 기존 감사 로그/다운로드 로그의 데이터 구조 변경

## 접근안

### 1. 권장안

새 `login_audit_logs` 테이블을 추가하고, 로그인 성공 시점에만 row를 적재한다. 관리자 카드 레이아웃은 `2 x 3`으로 고정하고, 로그형 카드 3개는 모두 기본 5개만 보여주며 `더 보기`로 5개씩 확장한다.

장점:

- 감사 목적에 맞다
- 로그 노이즈가 적다
- 운영 중인 기존 기능에 미치는 영향이 작다

단점:

- Supabase migration 1건이 추가된다

### 2. 서버 로그만 기록하는 안

로그인 성공을 서버 로그에만 남기고, 관리자 UI에는 표시하지 않는다.

장점:

- 구현이 더 작다

단점:

- 운영자 화면에서 직접 확인할 수 없다
- 이번 요구사항을 충족하지 못한다

## 선택

`1. 권장안`으로 진행한다.

## 데이터 설계

새 테이블: `public.login_audit_logs`

저장 필드:

- `id`
- `organization_id`
- `user_id`
- `user_email`
- `user_role`
- `ip_address`
- `user_agent`
- `created_at`

원칙:

- 로그인 성공 시점에만 적재
- 로그인 실패, 세션 확인, 토큰 refresh는 적재하지 않음
- 감사 로그 저장 실패가 로그인 자체를 막아서는 안 됨

## 백엔드 설계

### 저장 시점

명시적 로그인 성공 경로에서 세션/사용자 정보를 확정한 뒤, `login_audit_logs`에 best-effort insert를 수행한다.

### 저장 내용

- 조직 ID
- 로컬 사용자 ID
- 이메일
- 역할
- 요청 IP
- 요청 user-agent
- 생성 시각

### 조회 API

관리자 전용 API를 추가한다.

- `GET /api/admin/login-audit-logs?limit=N`

동작:

- 최근순 정렬
- 기본 `limit=5`
- 프론트의 `더 보기`는 limit를 `10`, `15`, `20`처럼 늘려 다시 조회

오류 처리:

- 저장 실패는 로그인 흐름을 깨지 않음
- 조회 실패는 관리자 카드에서 오류 메시지로만 표시

## 프론트 설계

### 카드 배치

관리자 카드 영역을 `2열 x 3행`으로 고정한다.

배치:

1행 왼쪽: 사용자 초대
1행 오른쪽: 사용자 상태 및 목록
2행 왼쪽: 조직 감사 로그
2행 오른쪽: 다운로드 이력
3행 왼쪽: 로그인 이력
3행 오른쪽: 빈 슬롯

오른쪽 아래 빈 슬롯은 의도적으로 비워 둔다. 추후 운영자 전용 카드가 필요해질 때만 재사용한다.

### 로그 카드 공통 규칙

대상:

- 조직 감사 로그
- 다운로드 이력
- 로그인 이력

공통 동작:

- 기본 5개 표시
- `더 보기` 버튼 클릭 시 5개씩 추가
- 더 이상 항목이 없으면 버튼 숨김 또는 비활성화
- 긴 텍스트는 카드 안에서 줄바꿈

### 로그인 이력 카드 내용

각 항목은 다음 정보를 보여준다.

- 사용자 이메일
- 역할
- 로그인 시각
- IP 주소
- user-agent 요약 또는 원문

운영자 모드가 아니면 렌더하지 않는다.

## 테스트 설계

백엔드:

- 로그인 성공 시 로그 row 적재
- 저장 실패가 로그인 응답을 깨지 않음
- 관리자 전용 조회 API 권한 확인
- `limit` 동작 확인

프론트:

- 관리자 모드에서만 로그인 이력 카드 노출
- 기본 5개만 노출
- `더 보기` 클릭 시 5개 증가
- 빈 상태 / 오류 상태 렌더
- `2 x 3` 배치에서 카드가 레이아웃을 깨지 않음

## 운영 적용 순서

1. 새 feature 브랜치에서 구현
2. 로컬 테스트 및 merge-result 검증
3. 코드 병합 준비
4. 마지막에 Supabase SQL Editor로 `login_audit_logs` migration 적용
5. 운영 배포 후 실제 로그인 1건과 다운로드/감사 카드 동작 확인

## 리스크와 완화

리스크:

- 로그인 경로에 감사 로그 insert가 추가되면서 인증 흐름이 느려지거나 깨질 수 있음
- 관리자 로그 카드가 너무 길어져 다시 레이아웃이 무너질 수 있음

완화:

- 로그인 감사 로그 insert는 best-effort로 수행
- 프론트는 기본 5개 노출과 `더 보기`로 길이를 제한
- 카드 레이아웃은 `2 x 3` 고정과 줄바꿈 규칙으로 유지
