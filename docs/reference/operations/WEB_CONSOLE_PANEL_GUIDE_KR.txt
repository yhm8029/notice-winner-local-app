웹 콘솔 패널 가이드

- 문서 역할: 화면/패널 참고
- 정본 여부: `reference`
- 이 문서가 답하는 질문: 웹 콘솔에 어떤 패널이 있고 각 패널이 무엇을 보여주는지
- 이 문서가 답하지 않는 질문: 제품 요구사항의 최종 정의, API/DB/정책의 최종 계약
- 상위 기준 문서: [../../00_CANONICAL_INDEX_KR.md](../../00_CANONICAL_INDEX_KR.md)
- 충돌 시 우선 문서: [../../00_CANONICAL_INDEX_KR.md](../../00_CANONICAL_INDEX_KR.md)

기준: notice-winner-pipeline-web 현재 프론트엔드 코드(index.html + frontend/app.js)

목적
- 콘솔에 보이는 각 패널이 왜 있는지, 무엇을 보여주는지, 어떤 버튼과 입력이 어떤 동작을 하는지 정리한다.
- 운영자가 실제로 매일 쓰는 기능인지, 상황에 따라 쓰는 보조 기능인지, 테스트/검증용 기능인지 구분한다.

분류 기준
- 실사용 필수
  - Phase 1 운영자가 실제 업무를 처리할 때 직접 쓰는 기능
- 상황별 사용
  - 항상 쓰는 것은 아니지만, 확인/재검색/수정/다운로드처럼 운영 중 자주 필요할 수 있는 기능
- 테스트·검증용
  - GUI parity 비교, 리포트 검증, 개발 확인처럼 운영보다 점검 목적이 강한 기능

0. 상단 헤더 / 동기화 바
================================================================================
패널명
- 상단 헤더
- API / 모드 / 자동 새로고침 / 마지막 동기화 시각

분류
- 상황별 사용

왜 있는가
- 지금 보고 있는 콘솔이 어느 API에 붙어 있는지 보여주기 위해 있다.
- 자동 새로고침이 켜져 있는지, 마지막으로 언제 화면이 갱신됐는지 바로 확인하게 하려는 영역이다.

무엇을 보여주는가
- API base label
- 현재 모드 문구
- 자동 새로고침 체크박스
- 마지막 동기화 시각

어떤 기능이 있는가
- 자동 새로고침 ON/OFF
- 현재 콘솔이 최신 상태인지 확인

세부 기능 설명
- API
  - 현재 콘솔이 어떤 백엔드 주소를 보고 있는지 보여준다.
  - 로컬 서버인지, 배포 서버인지 구분할 때 쓴다.
- 모드
  - 현재 콘솔이 어떤 운영 모드를 전제로 하고 있는지 보여주는 안내 영역이다.
  - 보통 운영 단계나 콘솔 성격을 설명하는 표시 역할을 한다.
- 자동 새로고침
  - 체크하면 대시보드, 실행 목록, 로그, 트래커 목록 등 주요 데이터를 주기적으로 다시 불러온다.
  - 실시간에 가깝게 상태를 보고 싶을 때 켠다.
  - 정지 상태에서 화면이 자꾸 바뀌는 게 불편하면 끌 수 있다.
- 마지막 동기화 시각
  - 화면 데이터가 마지막으로 언제 갱신됐는지 보여준다.
  - “지금 보는 값이 방금 불러온 것인지” 확인할 때 쓴다.

실제로 언제 쓰는가
- 서버 연결 상태를 확인할 때
- 새 실행이 끝났는지 기다릴 때
- 화면 값이 오래된 것 아닌지 확인할 때

1. 대시보드
================================================================================
패널명
- 대시보드

분류
- 실사용 필수

왜 있는가
- 운영 전체 상태를 한 화면에서 빠르게 요약해서 보여주기 위해 있다.
- “지금 시스템이 정상인지”, “실패 run이 있는지”, “트래커 데이터가 얼마나 쌓였는지”를 먼저 보는 영역이다.

무엇을 보여주는가
- Runs
- Running
- Failed
- Tracker Rows
- Edited Rows
- Artifacts persistence 상태
- 최신 검증 리포트 요약
- 최근 실패 실행 목록
- 실행 중인 리포트 작업 목록

어떤 기능이 있는가
- 대시보드 새로고침
- 최근 실패 실행 확인
- 리포트 작업 진행 상태 확인

세부 기능 설명
- 대시보드 새로고침
  - `/api/dashboard/summary`를 다시 호출해서 최신 운영 현황을 불러온다.
  - 자동 새로고침이 꺼져 있을 때 수동으로 최신 상태를 보고 싶으면 누른다.
- Runs
  - 전체 실행 건수를 보여준다.
  - run이 얼마나 누적됐는지 보는 지표다.
- Running
  - 현재 진행 중인 실행 건수를 보여준다.
  - 서버가 지금 몇 건을 처리 중인지 파악할 때 쓴다.
- Failed
  - 실패한 실행 수를 보여준다.
  - 운영 이상 징후를 가장 빨리 확인하는 지표다.
- Tracker Rows
  - 현재 저장된 트래커 row 수를 보여준다.
  - 실데이터가 얼마나 누적됐는지 보는 지표다.
- Edited Rows
  - 웹에서 재정의된 tracker row 수를 보여준다.
  - 수동 수정이 어느 정도 누적됐는지 보는 지표다.
- Artifacts persistence
  - 산출물 메타데이터가 영속 저장인지, 재시작 시 사라질 수 있는 상태인지 보여준다.
  - 운영 안정성 확인용이다.
- 최근 실패 실행
  - 실패한 run 일부를 카드 형태로 보여준다.
  - run id, 상태, 유형 등을 보고 어떤 실행이 실패했는지 바로 파악할 수 있다.
- 실행 중인 리포트 작업
  - GUI parity 비교 같은 검증 리포트가 아직 돌아가는지 보여준다.
  - 운영 필수라기보다 검증 작업을 병행할 때 유용하다.

실제로 언제 쓰는가
- 서버를 띄운 직후 전체 상태를 볼 때
- 실패 run 유무를 먼저 확인할 때
- 산출물 메타 저장 방식이 운영 가능한 상태인지 볼 때

2. 프로젝트 트랙커 실행 생성
================================================================================
패널명
- 프로젝트 트랙커 실행 생성

분류
- 실사용 필수

왜 있는가
- 새 `project_tracker` 부모 run을 생성하는 시작점이기 때문이다.
- 운영자가 실제 공고 검색 조건을 넣고 수집 파이프라인을 시작하는 핵심 패널이다.

무엇을 보여주는가
- 시작일 / 종료일 / 계약일 힌트
- 공고 번호
- 공고명
- 수요기관
- 페이지당 행 수 / 최대 페이지 수
- API 범위
- 고급 옵션

어떤 기능이 있는가
- 실행 시작
- 초기화
- 검색 조건 입력
- 수집 모드 선택
- LLM 보정 관련 옵션 입력

세부 기능 설명
- 실행 시작
  - 현재 폼에 입력한 조건으로 새 `project_tracker` 실행을 시작한다.
  - 내부적으로 `POST /api/runs`를 호출해 부모 run 1건을 만든다.
  - 실행이 시작되면 서버는 `collect -> filter -> rescan -> export -> finalize`를 순서대로 진행한다.
  - 이 버튼은 트래커 엑셀을 바로 만드는 버튼이 아니라, 원본 수집/가공 파이프라인을 시작하는 버튼이다.

- 초기화
  - 현재 폼 입력값을 화면 기본값으로 되돌린다.
  - 날짜, 검색어, 기관명, 페이지 수, 고급 옵션을 초기 상태로 리셋한다.
  - 이미 서버에 저장된 run, 로그, 아티팩트, tracker 데이터는 지우지 않는다.

- 검색 조건 입력
  - 무엇을 찾을지와 어디까지 찾을지를 정하는 입력 영역이다.
  - 시작일 / 종료일
    - 조회 기간을 지정한다.
    - 서버에는 `YYYYMMDD` 형식으로 전달된다.
  - 계약일 힌트
    - 후속 계약 조회 보정에 참고하는 날짜 힌트다.
    - 비워두면 기본 규칙으로 처리된다.
  - 공고 번호
    - 특정 공고 1건을 직접 겨냥할 때 쓰는 입력값이다.
    - 가장 강한 직접 검색 조건이다.
  - 공고명
    - 제목 키워드로 공고를 찾을 때 쓰는 입력값이다.
  - 수요기관
    - 기관명 키워드로 공고를 좁히는 입력값이다.
  - 페이지당 행 수 / 최대 페이지 수
    - 결과를 얼마나 많이 읽을지 정하는 범위 제한값이다.
  - API 범위
    - construction / service / goods / all 중 어디를 조회할지 정한다.

- 수집 모드 선택
  - 수집이 어떤 백엔드 경로를 탈지 정하는 옵션이다.
  - `auto`
    - 기본 운영 모드다.
    - 가능한 실제 수집 경로를 사용한다.
    - 실사용 기준 기본값이다.
  - `gui_seed`
    - GUI parity 비교용 시드 경로를 쓰는 모드다.
    - 실운영 기본 모드가 아니라, 비교/검증/이식 점검 용도가 강하다.
  - `synthetic`
    - 일반 운영에는 숨겨지는 디버그 모드다.
    - 특별한 디버그 플래그를 켠 경우에만 보이거나 동작한다.
    - 테스트·개발 전용으로 봐야 한다.

- LLM 보정 관련 옵션 입력
  - 문서 파싱 결과가 애매한 경우 후처리에서 LLM 보정을 추가로 적용할지 정하는 고급 옵션이다.
  - LLM 보정
    - 켜면 보정 로직을 사용할 수 있다.
    - 끄면 규칙 기반 결과만 사용한다.
  - LLM 모델
    - 어떤 모델 이름을 쓸지 지정한다.
  - LLM 최대 건수
    - 한 실행에서 LLM 보정을 적용할 최대 row 수를 제한한다.
  - 이 영역은 실사용에서 “항상 필요한 값”은 아니다.
  - 기본 추출 품질이 충분하면 꺼두고 써도 된다.
  - 정밀 보정이 필요할 때만 켜는 상황별 사용 기능이다.

이 패널이 실사용에 필요한가
- 예. 가장 핵심적인 실사용 패널이다.

이 패널 안에서 테스트/개발용 요소는 무엇인가
- `gui_seed`
- `synthetic`
- 단계 지연값 같은 디버그성 옵션

3. 실행 프리셋 카드
================================================================================
패널명
- 실행 프리셋

분류
- 상황별 사용

왜 있는가
- 자주 쓰는 검색 조건을 저장하고 다시 쓰게 하려는 보조 패널이다.
- 매번 동일한 기간, 동일한 기관명, 동일한 API 범위를 다시 타이핑하지 않게 하려는 목적이다.

무엇을 보여주는가
- 저장된 프리셋 목록
- 프리셋 상태 문구

어떤 기능이 있는가
- 새로고침
- 적용
- 현재 조건 저장

세부 기능 설명
- 새로고침
  - 저장된 프리셋 목록을 다시 불러온다.
  - 다른 사람이 저장했거나 방금 저장한 프리셋을 다시 확인할 때 쓴다.
- 적용
  - 선택한 프리셋의 조건을 실행 생성 폼에 채워 넣는다.
  - 바로 실행하지는 않는다.
  - 먼저 폼을 채우고, 필요하면 일부 수정한 뒤 실행 시작을 누른다.
- 현재 조건 저장
  - 지금 실행 생성 폼에 들어 있는 조건을 프리셋으로 저장한다.
  - 자주 반복되는 검색 조합을 재사용할 때 유용하다.

이 패널이 실사용에 필요한가
- 없어도 운영은 가능하다.
- 하지만 반복 검색이 많으면 매우 유용한 실사용 보조 기능이다.

이 패널이 테스트/개발용인가
- 아니오.
- 운영 편의 기능이다.

4. 실행 상세
================================================================================
패널명
- 실행 상세

분류
- 실사용 필수

왜 있는가
- 선택한 run 1건의 상태, 진행률, 요청값, 요약값, 오류값을 한 곳에서 보게 하기 위해 있다.
- 운영자가 현재 실행이 어디까지 갔는지 판단하는 중심 패널이다.

무엇을 보여주는가
- 실행 ID
- 상태 배지
- 진행률
- run type
- 현재 단계
- 요청 JSON
- 요약 JSON
- 오류 JSON
- 생성 / 시작 / 종료 시각
- 상위 실행 ID

어떤 기능이 있는가
- 새로고침
- 실행 취소
- 트래커 내보내기 생성

세부 기능 설명
- 새로고침
  - 현재 선택한 run 상세를 다시 불러온다.
  - 자동 새로고침이 꺼져 있거나 즉시 반영이 필요할 때 쓴다.
- 실행 취소
  - 아직 끝나지 않은 run에 취소 요청을 보낸다.
  - 내부적으로 `POST /api/runs/{run_id}/cancel`을 호출한다.
  - 바로 결과가 종료되는 것이 아니라 다음 체크 지점에서 `cancelled`로 반영된다.
- 트래커 내보내기 생성
  - 성공한 부모 `project_tracker` run에서 child `tracker_export` 실행을 새로 만든다.
  - 즉 “수집 결과를 트래커 엑셀/트래커 row로 만드는 다음 단계” 버튼이다.
  - 부모 run이 성공 상태가 아니면 쓸 수 없다.

이 패널이 실사용에 필요한가
- 예. 운영 중 가장 자주 보는 핵심 패널이다.

이 패널 안에서 테스트/개발용 요소는 무엇인가
- 요청 JSON, 요약 JSON, 오류 JSON 자체는 운영에도 필요하지만, 개발 디버깅 성격도 강하다.

5. 실행 경로 / 실행 컨텍스트 카드
================================================================================
패널명
- 실행 경로
- 실행 컨텍스트

분류
- 상황별 사용

왜 있는가
- 현재 run이 실제로 어떤 백엔드 경로를 탔는지 설명하기 위해 있다.
- native로 갔는지, fallback이 있었는지, 어떤 stage backend가 선택됐는지 빠르게 보기 위한 카드다.

무엇을 보여주는가
- 요청 collect mode
- 실제 collect backend
- stage_backends collect/filter/rescan/export
- quota fallback 여부
- runtime profile

- 읽기 전용 확인

세부 기능 설명

- 실제 collect backend
  - 실제로 어떤 백엔드가 실행됐는지 보여준다.

- 토스트 메시지

분류
- 상황별 사용

왜 있는가
- 저장 성공, API 실패, 실행 생성 완료 같은 즉시 결과를 사용자에게 바로 알려주기 위해 있다.

무엇을 보여주는가
- 성공 / 오류 / 경고 메시지

어떤 기능이 있는가
- 읽기 전용 알림

세부 기능 설명
- 실행 생성 성공 알림
  - run이 큐에 들어갔는지 즉시 알려준다.
- 저장 성공 알림
  - tracker override 저장이 완료됐는지 보여준다.
- 오류 알림
  - validation error나 API 실패를 즉시 보여준다.

이 패널이 실사용에 필요한가
- 보조적이지만 중요하다.
- 운영자가 버튼 결과를 즉시 인지하게 해준다.

이 패널이 테스트/개발용인가
- 아니다.
- 운영 UX 보조 기능이다.

================================================================================
실사용 / 상황별 / 테스트용 요약
================================================================================
실사용 필수
- 대시보드
- 프로젝트 트랙커 실행 생성
- 실행 상세
- 최근 실행
- 로그
- 아티팩트
- 트래커 항목
- 트래커 항목 수정

상황별 사용
- 상단 헤더 / 동기화 바
- 실행 프리셋 카드
- 실행 경로 / 실행 컨텍스트 카드
- 프로젝트 집계 카드
- 트래커 결과 보드
- 트래커 항목 드로어
- 플래시 메시지 / 토스트

테스트·검증용
- 최신 리포트
- 실행 생성 패널 안의 `gui_seed`
- 실행 생성 패널 안의 숨겨진 `synthetic` 디버그 모드

================================================================================
운영자가 실제로 많이 쓰는 흐름
================================================================================
1. 실행 생성 패널에서 검색 조건 입력 후 실행 시작
2. 최근 실행에서 새 `project_tracker` 선택
3. 실행 상세와 로그로 완료 여부 확인
4. 필요하면 아티팩트에서 `winner_csv` 미리보기 또는 다운로드
5. 부모 run 성공 후 `트래커 내보내기 생성` 실행
6. child `tracker_export` 선택
7. 아티팩트에서 `tracking_excel` 확인
8. 트래커 항목 / 트래커 결과 보드에서 row 확인
9. 필요하면 트래커 항목 수정 패널에서 override 저장
10. 감사 로그로 누가 무엇을 바꿨는지 확인
