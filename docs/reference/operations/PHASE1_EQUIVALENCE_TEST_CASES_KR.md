# Phase 1 동등성 테스트 케이스

- 문서 역할: 검증 기준 참고
- 정본 여부: `reference`
- 이 문서가 답하는 질문: Phase 1 parity를 어떤 시나리오로 검증할지, 완료판정 전에 어떤 테스트를 볼지
- 이 문서가 답하지 않는 질문: 제품 요구사항의 최종 정의, 시스템 설계의 최종 구조

- 충돌 시 우선 문서: [../../00_CANONICAL_INDEX_KR.md](../../00_CANONICAL_INDEX_KR.md)

## 업데이트 메모
- 추가 업데이트: 2026-03-13
- 이번 업데이트에서 `동등성 검증 통과 = Phase 1 완료`가 아님을 다시 명시함
- 이번 업데이트에서 `트래커 핵심 필드 충실도`를 별도 완료 조건으로 추가함

## 목적
- 웹 구현이 기존 GUI와 결과 흐름상 얼마나 가까운지 검증한다.
- 이 문서는 `검증 기준` 문서이지, `완료 선언 문서`가 아니다.

## 공통 원칙
- 같은 입력이면 성공/실패 판단이 유사해야 한다.
- 상태 전이는 유사해야 한다.
- 주요 로그 흐름은 유사해야 한다.
- 필수 산출물 생성 여부는 유사해야 한다.
- 단, 아래 항목이 비어 있으면 동등성 일부가 맞아도 Phase 1 완료가 아니다.
  - 트래커 핵심 필드 충실도
  - 웹에서의 트래커 결과 확인성

## 비교 대상
- 입력값 검증
- 실행 상태: `queued | running | success | failed | cancelled`
- 로그 흐름
- `winner_csv` 생성 여부
- `tracking_excel` 생성 여부
- tracker export 성공/실패 처리

## 추가 완료 조건
아래 조건은 동등성 테스트 케이스와 별도로 만족해야 한다.
1. 트래커 엑셀의 주요 컬럼이 실제 데이터로 채워져야 한다.
2. 웹에서도 트래커 양식 결과를 충분히 확인할 수 있어야 한다.
3. 임시 placeholder 값이 남아 있으면 안 된다.

## 테스트 케이스 1. 기본 정상 실행
### 목적
- 일반적인 기간 검색 실행이 정상 완료되는지 검증한다.

### 기대 결과
- 입력 검증 통과
- 상태 전이: `queued -> running -> success`
- `winner_csv` 생성

## 테스트 케이스 2. 입력 검증 실패
### 목적
- 입력 검증 규칙이 적용되는지 확인한다.

### 기대 결과
- validation error 반환
- run 미생성

## 테스트 케이스 3. 실행 중 취소
### 목적
- 취소 요청이 반영되는지 확인한다.

### 기대 결과
- 최종 상태 `cancelled`
- 취소 관련 로그 존재

## 테스트 케이스 4. fallback 또는 수집 실패 처리
### 목적
- 수집 누락/재시도/실패 상황에서 상태와 로그가 적절한지 확인한다.

### 기대 결과
- 실패 또는 재시도 경로가 명확히 남는다.
- fallback이 있으면 그 근거가 남는다.

## 테스트 케이스 5. tracker_export
### 목적
- `project_tracker` 결과로 child run 기반 tracker export가 생성되는지 검증한다.

### 기대 결과
- child run 생성
- `tracking_excel` 생성
- child run 성공/실패가 부모 run 상태를 오염시키지 않음

## 주의
- 위 5개 케이스 통과는 중요하다.
- 하지만 이것만으로 `GUI 기능 100% 이관 완료`라고 판단하지 않는다.
- 특히 트래커 핵심 필드가 빈 값이거나 placeholder 값이면 Phase 1 미완료다.
