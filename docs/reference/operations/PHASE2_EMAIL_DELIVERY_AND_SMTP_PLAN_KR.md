## 1. 문서 목적

이 문서는 Phase 2에서 사용하는 초대 메일, 가입 메일, 비밀번호 재설정 메일의 발송 전략을 정리한다.

현재 구현은 `Supabase Auth`를 기준으로 초대/가입 흐름을 구성하고 있으나, 메일 발송 채널은 단계적으로 운영한다.

## 2. 현재 결정

현재 결정은 아래와 같다.

1. 초기 운영은 `Gmail SMTP`를 사용한다.
2. 이후 운영 안정화 단계에서 `Resend`로 전환한다.
3. 최종 운영 발신 주소는 개인 메일이 아니라 `회사 도메인 기반 no-reply 주소`를 목표로 한다.

## 3. 왜 지금 Resend를 바로 붙이지 않는가

현재 `panotec.co.kr` 도메인은 즉시 메일 발송용 서브도메인 검증을 진행하기 어려운 상태다.

정리:

1. 회사 도메인 또는 서브도메인 검증이 아직 즉시 가능하지 않다.
2. 따라서 개발/초기 테스트 단계에서는 `Gmail SMTP`가 가장 빠르다.
3. 다만 이 방식은 임시 운영 전략이며, 장기 운영 전략이 아니다.

## 4. 현재 임시 운영안

### 4.1 메일 발송 채널

- `Supabase Auth + Custom SMTP`
- SMTP 공급자: `Gmail SMTP`

### 4.2 적용 대상

- 사용자 초대 메일
- 회원가입/초대 수락 메일
- 비밀번호 재설정 메일

### 4.3 현재 임시 발신자 정책

- 발신 주소는 Gmail 계정 기준으로 운영한다.
- 즉, 수신자에게는 개인 Gmail 주소 또는 Gmail 기반 발신 주소가 보일 수 있다.

이 단계에서는 아래를 감수한다.

1. 발신자 신뢰도가 회사 도메인 기반보다 낮다.
2. 일일 발송량 제한이 있다.
3. 대량 운영에는 맞지 않는다.

## 5. Gmail SMTP 임시 운영 제약

Gmail SMTP를 임시로 쓰는 이유는 빠른 테스트/초기 운영 때문이다. 하지만 아래 제약이 있다.

1. 개인 Gmail은 대량 트랜잭션 메일에 적합하지 않다.
2. 발신 주소가 `no-reply@company.com`처럼 자연스럽지 않을 수 있다.
3. 수신자에게 개인 메일 또는 Gmail 기반 발신자로 보일 수 있다.
4. 발송량이 늘어나면 일일 한도에 걸릴 수 있다.

따라서 Gmail SMTP는 `개발/초기 운영용 임시 채널`로만 사용한다.

## 6. 향후 목표 운영안

### 6.1 목표 채널

- `Resend`

### 6.2 목표 발신 도메인

예시:

- `auth.panotec.co.kr`
- 발신 주소: `no-reply@auth.panotec.co.kr`

### 6.3 목표 효과

1. 초대/가입/재설정 메일의 발신자를 회사 서비스 이름으로 통일한다.
2. 메일 평판을 회사 도메인 또는 인증용 서브도메인 기준으로 관리한다.
3. 초기 Gmail SMTP보다 높은 발송량을 안정적으로 처리한다.
4. 운영자가 개인 Gmail 주소를 직접 노출하지 않도록 한다.

## 7. Resend 전환 조건

아래 조건 중 하나라도 충족되면 Gmail SMTP에서 Resend로 전환한다.

1. 일일 발송량이 Gmail 임시 운영 범위를 넘기 시작한다.
2. 외부 고객사/실사용자 대상 초대 메일을 본격 발송해야 한다.
3. 발신자 주소를 `회사 도메인 기반 no-reply` 형태로 정식 운영해야 한다.
4. 메일 도달률, SPF/DKIM, 발신자 신뢰도 관리가 중요해진다.

## 8. Resend 전환 시 작업 항목

1. `Resend` 계정 준비
2. 메일 발송용 도메인 또는 서브도메인 검증
3. DNS에 SPF/DKIM 반영
4. Supabase Auth의 Custom SMTP를 Resend SMTP로 변경
5. 발신 주소를 `no-reply@auth.panotec.co.kr` 형태로 변경
6. 초대/비밀번호 재설정/가입 메일을 재테스트

## 9. 제품/운영 문구 기준

현재 문서/제품에서 메일 발송 전략을 설명할 때는 아래처럼 정리한다.

- 현재: `Gmail SMTP 기반 임시 운영`
- 향후: `Resend + 회사 도메인 기반 no-reply 발신`

운영자용 설명:

> 개발/초기 운영 단계에서는 Gmail SMTP를 사용한다.  
> 회사 도메인 검증이 가능한 시점에 Resend로 전환하고, 발신 주소는 회사 도메인 기반 no-reply로 고정한다.

## 10. 구현 원칙

1. 애플리케이션 코드는 특정 메일 공급자에 강하게 묶이지 않도록 유지한다.
2. 실제 인증/초대 메일 발송은 `Supabase Auth + Custom SMTP` 레이어에서 처리한다.
3. 공급자 교체는 `SMTP 설정 변경` 중심으로 끝낼 수 있게 한다.
4. 메일 템플릿은 공급자 종속이 아니라 제품 정책 중심으로 유지한다.

## 11. 지금 하지 않는 것

현재 단계에서는 아래를 바로 하지 않는다.

1. Resend 직접 API 호출 방식으로 애플리케이션 코드 변경
2. 멀티 공급자 자동 failover
3. 메일 발송량 기반 자동 공급자 전환
4. 마케팅 메일/뉴스레터 시스템

## 12. 결론

현재 Phase 2 메일 전략은 아래와 같다.

1. 지금은 `Gmail SMTP`로 먼저 운영한다.
2. 이 방식은 개발/초기 운영용 임시 정책이다.
3. 장기 운영 기준은 `Resend + 회사 도메인 기반 no-reply 주소`다.
4. 도메인 검증 여건이 준비되면 `Resend`로 전환한다.
