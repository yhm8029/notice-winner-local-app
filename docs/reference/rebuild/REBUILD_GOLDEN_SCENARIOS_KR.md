# 재구축 골든 시나리오

- 문서 역할: 재구축 검증용 골든 시나리오
- 정본 여부: `reference`
- 이 문서가 답하는 질문: 재구축된 시스템이 현재 구현과 충분히 유사한지 무엇으로 검증하는가
- 이 문서가 답하지 않는 질문: 최종 계약 우선순위, 전체 API 필드 사전, 모든 DB 제약
- 상위 기준 문서: [../.../../spec/REBUILD_IMPLEMENTATION_PLAYBOOK_KR.md](../.../../spec/REBUILD_IMPLEMENTATION_PLAYBOOK_KR.md)
- 충돌 시 우선 문서: [../../00_CANONICAL_INDEX_KR.md](../../00_CANONICAL_INDEX_KR.md), [../.../../spec/FUNCTIONAL_SPEC_KR.md](../.../../spec/FUNCTIONAL_SPEC_KR.md), [../.../../spec/UI_SCREEN_SPEC_KR.md](../.../../spec/UI_SCREEN_SPEC_KR.md)

작성일: 2026-03-22  
목적: 문서만 보고 재구축한 결과가 현재 구현과 95% 이상 유사한지 검증할 대표 시나리오를 고정한다.

## 1. 사용 원칙

1. 이 문서는 구현 순서를 결정하지 않는다.
2. 이 문서는 재구축 완료 후 `무엇을 실제로 눌러보고 맞는지`를 검증하는 기준이다.
3. 각 시나리오는 `시작 상태`, `행동`, `예상 결과`, `실패 기준`을 함께 가진다.

## 2. 시나리오 1: 최초 운영자 로그인

### 시작 상태

- `bootstrap platform_admin` 이메일이 환경에 등록되어 있다.
- 아직 로그인 세션이 없다.

### 행동

1. `/app/` 접속
2. 로그인 화면 확인
3. bootstrap 이메일과 비밀번호로 로그인

### 예상 결과

1. 로그인 shell이 보인다.
2. 로그인 성공 후 콘솔이 열린다.
3. 관리자 모드 전환이 가능하다.
4. 세션 정보에 `platform_admin` 성격 정보가 반영된다.

### 실패 기준

- 로그인 후에도 콘솔이 아닌 차단 화면에 머문다.
- 관리자 모드 진입이 안 된다.

## 3. 시나리오 2: 초대 생성과 수락

### 시작 상태

- `platform_admin` 또는 `org_admin`이 로그인 상태다.
- 초대 대상 이메일은 현재 로그인 이메일과 다르다.

### 행동

1. `사용자 초대 및 관리` 진입
2. 이메일, 표시 이름, 역할, 팀명, 직책 입력
3. 초대 링크 생성
4. 초대 링크 또는 메일로 수락 페이지 진입
5. 초대 이메일과 같은 이메일로 로그인 또는 가입

### 예상 결과

1. 초대가 `pending` 상태로 생성된다.
2. 초대 이메일과 다른 이메일로는 수락이 차단된다.
3. 같은 이메일로 수락 시 membership이 생성되거나 연결된다.
4. 초대 상태는 `accepted`로 바뀐다.

### 실패 기준

- 자기 자신 초대가 허용된다.
- 다른 이메일로 수락이 된다.
- 같은 초대를 두 번 눌러 membership이 중복 생성된다.

## 4. 시나리오 3: 회원정보 수정

### 시작 상태

- 일반 사용자 또는 관리자 계정으로 로그인되어 있다.

### 행동

1. `회원정보` 열기
2. 표시 이름, 휴대폰, 회사 전화 수정
3. 현재 비밀번호 입력
4. 저장

### 예상 결과

1. 현재 비밀번호 없이 저장되지 않는다.
2. 저장 후 상단 사용자 표시와 사용자 목록에 최신 값이 반영된다.
3. 이메일과 회사명은 읽기 전용이다.

### 실패 기준

- 현재 비밀번호 없이 프로필이 바뀐다.
- 수정 후 다시 열었을 때 값이 원복된다.

## 5. 시나리오 4: 실행 생성과 tracker export

### 시작 상태

- 권한 있는 사용자가 로그인되어 있다.

### 행동

1. 실행 생성 폼 입력
2. `project_tracker` 실행 시작
3. run 상세에서 진행률, 로그, 상태 확인
4. tracker export child run 생성 또는 자동 실행 확인
5. tracker workbook 다운로드

### 예상 결과

1. 부모 run과 child run이 분리되어 보인다.
2. child 실패가 부모 상태를 직접 뒤집지 않는다.
3. workbook 다운로드가 가능하다.

### 실패 기준

- child run이 부모 run처럼 덮여서 보인다.
- 다운로드 버튼이 success 상태인데도 동작하지 않는다.

## 6. 시나리오 5: tracker 수정과 감사 로그

### 시작 상태

- tracker entries가 최소 1건 이상 있다.

### 행동

1. 관리자 모드에서 tracker row 선택
2. 필드 수정 저장
3. audit log 확인

### 예상 결과

1. effective tracker 값이 즉시 갱신된다.
2. audit log에 actor, before/after, 시각이 남는다.
3. 같은 화면 또는 새로고침 후에도 수정값이 유지된다.

### 실패 기준

- 수정은 됐는데 audit log가 남지 않는다.
- 화면 값만 바뀌고 실제 재조회하면 원복된다.

## 7. 시나리오 6: 영업 시작과 사용자 홈 분기

### 시작 상태

- 영업 대상 프로젝트가 최소 1건 있다.
- 일반 사용자 계정으로 로그인되어 있다.

### 행동

1. `전체 영업 대상 프로젝트`에서 `영업` 클릭
2. `내가 진행 중인 영업` 이동 확인
3. `회사 전체 진행 중인 영업` 반영 확인

### 예상 결과

1. 같은 프로젝트가 `전체 영업 대상 프로젝트`에서 사라진다.
2. `내가 진행 중인 영업`에서 편집 가능 상태로 나타난다.
3. `회사 전체 진행 중인 영업`에는 읽기 전용으로 나타난다.

### 실패 기준

- 영업 시작 후에도 대상 목록에 남아 있다.
- 회사 전체 영역에서 수정이 가능하다.

## 8. 시나리오 7: 영업 메모 추가

### 시작 상태

- 로그인한 사용자가 claim owner다.

### 행동

1. `내가 진행 중인 영업`에서 메모 입력
2. 저장

### 예상 결과

1. 메모가 append-only로 누적된다.
2. 날짜는 KST 기준 `YYYY-MM-DD(요일)` 형식으로 보인다.
3. 같은 내용이 `회사 전체 진행 중인 영업` 읽기 모델에도 반영된다.

### 실패 기준

- 저장 후 이전 메모가 사라진다.
- 회사 전체 영역과 내용이 불일치한다.

## 9. 시나리오 8: 담당 이관과 비활성화 차단

### 시작 상태

- 사용자 A가 프로젝트를 진행 중이다.
- 관리자 계정이 있다.

### 행동

1. 사용자 A가 사용자 B로 담당 이관
2. 또는 관리자가 강제 이관
3. 진행 중 영업이 남은 상태에서 사용자 비활성화 시도

### 예상 결과

1. 이관 후 owner가 B로 바뀐다.
2. 기존 메모는 유지된다.
3. 진행 중 영업이 남은 사용자는 바로 비활성화되지 않는다.

### 실패 기준

- 이관 시 메모 이력이 사라진다.
- 진행 중 영업이 있는데도 계정이 바로 비활성화된다.

## 10. 시나리오 9: 계약 완료

### 시작 상태

- 로그인한 사용자가 claim owner다.

### 행동

1. `계약 완료` 클릭
2. 계약금액 입력 모달 표시
3. 숫자만 입력, 콤마 자동 포맷 확인
4. 저장

### 예상 결과

1. 계약금액 없이 저장되지 않는다.
2. 상태가 `계약 완료`로 바뀐다.
3. `내가 진행 중인 영업`, `회사 전체 진행 중인 영업`, `전체 영업 대상 프로젝트`에서 빠진다.
4. `종료/완료 정리`의 연도/월 그룹에 들어간다.

### 실패 기준

- 계약금액 없이 완료 처리된다.
- 종료/완료 정리에 나타나지 않는다.

## 11. 시나리오 10: 영업 종료

### 시작 상태

- 로그인한 사용자가 claim owner다.

### 행동

1. `영업 종료` 클릭
2. 종료 처리

### 예상 결과

1. 상태가 `영업 종료`로 바뀐다.
2. 진행 중 영역에서는 사라진다.
3. 종료/완료 정리에 `영업 종료`로 나타난다.

### 실패 기준

- 종료 후에도 진행 중 영역에 남아 있다.

## 12. 시나리오 11: 사용자 모드/관리자 모드 UI 분기

### 행동

1. 일반 사용자로 로그인
2. 관리자 모드 전환 시도
3. 관리자 계정으로 로그인
4. 관리자 모드 전환

### 예상 결과

1. 일반 사용자 모드에선 실행 상세, 사용자 관리 같은 운영 패널이 숨겨진다.
2. 관리자만 `사용자 초대 및 관리`, 실행 상세, 영업 집계가 보인다.

### 실패 기준

- 일반 사용자에게 관리자 패널이 노출된다.

## 13. 시나리오 12: 엑셀 다운로드

### 행동

1. `내가 진행 중인 영업` 엑셀 다운로드
2. `회사 전체 진행 중인 영업` 엑셀 다운로드
3. tracker workbook 다운로드

### 예상 결과

1. 로컬 템플릿 기반의 `.xlsx` 파일이 내려온다.
2. 목록 범위에 맞는 row만 포함된다.

### 실패 기준

- 버튼은 보이는데 파일이 내려오지 않는다.
- 목록과 다른 데이터가 엑셀에 담긴다.

## 14. 최종 판정

아래를 모두 만족하면 `문서 기반 재구축이 현재 구현과 충분히 유사하다`고 본다.

1. 인증/권한/초대가 동작
2. 실행/트래커/workbook이 동작
3. 사용자/관리자 UI 분기가 맞다
4. 영업 claim/메모/이관/완료/종료가 맞다
5. 다운로드와 종료 정리 화면이 맞다
