# Phase 2 플랫폼 레이어 구조도

## 문서 목적
- 이 문서는 Phase 2를 `로그인 기능 추가`가 아니라 `B2B 운영 플랫폼 구조 확장`으로 해석하기 위한 상위 구조 문서다.
- Supabase에 맡길 것과 우리 서비스가 직접 가져가야 할 것을 분리한다.
- 현재 구현 범위와 향후 확장 가능 범위를 같은 프레임으로 정리한다.

## 한 줄 정의
이 서비스의 목표는 `Supabase 같은 범용 BaaS를 만드는 것`이 아니라, `Supabase 위에 조직 운영과 설계공모/영업 도메인 기능을 얹은 B2B 운영 플랫폼`을 만드는 것이다.

## 3단 레이어 구조

### 1. 인프라 레이어
역할:
- 인증의 기본기
- DB 저장소
- 파일 저장소
- 세션/토큰 처리
- 기본 API 접근 채널

현재 기준 포함 대상:
- Supabase Auth
- Supabase Postgres
- Supabase Storage
- 기본 세션 발급/검증
- 기본 사용자 인증 정보

여기에 두는 이유:
- 이 영역은 범용 인프라 성격이 강하다.
- 직접 만들면 유지보수 비용과 보안 리스크가 크다.
- 우리 서비스의 차별점은 이 레이어가 아니라 상위 업무 레이어에 있다.

원칙:
- 가능하면 사서 쓴다.
- 직접 만들더라도 최소화한다.
- 업무 정책을 인프라에 과도하게 집어넣지 않는다.

### 2. 공통 운영 레이어
역할:
- 회사 단위 고객 운영
- 사용자 초대/활성화/비활성화
- 좌석 수 제한
- 역할 분리
- 감사 로그
- 관리자 콘솔
- 세션 정책

현재 또는 Phase 2에서 가져가야 하는 핵심 대상:
- `organization`
- `organization_membership`
- `organization_invitation`
- `active_user_limit / pending_invite_limit`
- `platform_admin / org_admin / org_member`
- 로그인 후 권한 판별
- 초대 기반 가입
- signed cookie + refresh 기반 세션 정책
- 사용자/회사/역할 기준 감사 로그

이 레이어의 의미:
- 제품을 `회사에 팔 수 있는 SaaS`로 만드는 골격이다.
- 공용 계정 돌려쓰기를 운영적으로 억제한다.
- 과금 단위와 권한 단위를 만든다.
- 이후 어떤 도메인 기능이 추가돼도 공통적으로 재사용된다.

원칙:
- 조직 단위로 고객을 관리한다.
- 계정은 `1인 1계정`을 기본 정책으로 둔다.
- 자유가입보다 `초대 기반 가입`을 기본값으로 둔다.
- 권한은 UI 표시뿐 아니라 API/데이터 접근 제어까지 포함한다.

### 3. 도메인 기능 레이어
역할:
- 실제 고객이 돈을 내는 이유가 되는 업무 기능
- 업종 특화 자동화
- 실무자/관리자 화면
- 보고/집계/운영 규칙

현재 서비스 기준 핵심 대상:
- 공고 수집/검색/필터링/재탐색
- 당선사/계약 정보 수집
- tracker export
- tracker 결과 보드
- tracker 필드 수정
- 관련 공고 연결
- 영업 점유/잠금
- 영업현황 메모
- 영업 우선순위
- 다음 액션과 목표일
- 영업 진행 상태
- 관리자용 영업사원별 집계
- 추정 금액 합계/범위 계산
- 프로젝트 파이프라인 상태 표시

이 레이어의 의미:
- 이 서비스의 진짜 상품성은 여기에 있다.
- 고객은 `로그인 기능` 자체가 아니라 `공고/트래커/영업 운영`이 되길 원한다.
- 이 레이어가 업종 전용 백오피스 플랫폼의 본체다.

원칙:
- 도메인 규칙은 여기에서 명시한다.
- 범용 CRUD가 아니라 `업무 상태 전이`와 `운영 규칙`을 우선한다.
- 감사 로그는 도메인 이벤트까지 남긴다.

## 책임 경계

### Supabase에 맡기는 것
- 이메일 로그인/비밀번호 로그인
- Google 로그인
- 사용자 인증 토큰
- 세션의 기본 유지
- Postgres 데이터 저장
- 파일 저장소
- 기본적인 사용자 인증 관련 API

### 우리 서비스가 직접 가져갈 것
- 조직 모델
- 초대 기반 가입 정책
- seat 정책
- 역할 정책
- 감사 로그 정책
- 관리자/회사 관리자/일반 사용자 화면 분리
- tracker 수정 규칙
- 영업 claim 규칙
- 본인 해제/관리자 강제 해제
- 관리자 집계
- 프로젝트 파이프라인 현황
- 승인 흐름, 후속 업무 규칙

### 이 경계를 유지해야 하는 이유
- 인프라까지 직접 만들면 제품이 아니라 플랫폼 회사가 된다.
- 반대로 도메인 규칙까지 Supabase에 기대면 사업 차별점이 사라진다.
- `인프라는 외부 서비스`, `운영 정책과 업무 규칙은 우리 코드`라는 경계가 가장 효율적이다.

## 역할 구조

### 플랫폼 관리자
역할:
- 서비스 전체 운영자
- 회사 생성/비활성화
- seat 설정
- 회사 관리자 지정
- 전역 감사 로그 조회
- 장애 대응

### 회사 관리자
역할:
- 자기 회사 사용자 초대/비활성화
- 자기 회사 seat 관리
- 자기 회사 tracker 수정
- 자기 회사 run 생성/취소
- 자기 회사 감사 로그 조회

### 회사 이용자
역할:
- 자기 회사 프로젝트/트래커 조회
- 허용된 범위 내 작업 수행
- 향후 정책에 따라 제한된 수정 또는 영업 claim 수행

## Phase 2의 실질적 목표
- 로그인 화면을 추가하는 것
- 콘솔 진입 전 인증을 요구하는 것
- 로그인한 사용자의 회사/역할을 식별하는 것
- 역할에 따라 보이는 메뉴와 가능한 액션을 분리하는 것
- 모든 주요 수정/실행/초대/권한 변경을 추적하는 것
- 회사 단위로 계약과 사용자를 운영할 수 있게 만드는 것

즉, Phase 2는 `웹 로그인 기능`이 아니라 `다중 회사/다중 사용자 운영 구조`다.

## 현재 포함 범위와 향후 확장 범위

### Phase 2 초기 범위
- Supabase Auth 연동
- bootstrap platform admin
- 초대 기반 가입의 뼈대
- organization / membership / limit 구조
- 권한별 콘솔 진입
- tracker와 영업 도메인에서 사용자 식별 반영
- 조직 감사 로그 조회 화면

### 중기 확장 범위
- org_admin 초대 화면
- 사용자 활성/비활성 UI
- 단일 활성 세션 hardening
- 영업 claim 영속 저장
- 영업 액션 최소 모델(`owner / priority / next_action / due_date / status / memo`)
- 승인 흐름

### 장기 확장 범위
- 회사 SSO
- 팀/부서 단위 세분화 권한
- 승인 체계 고도화
- 자동 알림
- 영업 파이프라인 분석
- 성과 집계/리포트
- 업종 전용 관리자 백오피스 확장

## 사업 관점 해석

### B2B용 Auth + 조직 + 초대 + seat + 감사로그
의미:
- 회사에 팔 수 있는 SaaS 운영 기반
- 계정 공유 억제
- 계약 단위와 사용자 단위 생성

### 내부업무툴용 DB 콘솔 + 승인 흐름 + 권한관리
의미:
- 실무자가 실제 데이터를 안전하게 다루는 운영 도구
- 단순 CRUD가 아니라 승인/해제/추적이 들어간 업무 시스템

### 특정 업종 전용 백오피스 플랫폼
의미:
- 설계공모/영업/프로젝트 추적에 특화된 운영 플랫폼
- 범용 CRM이나 범용 데이터베이스 UI와 다른 상품성

## 설계 원칙
- 인프라는 재사용하고, 업무 규칙은 직접 만든다.
- 인증은 외부 서비스에 기대되, 권한 의미는 우리 서비스가 정의한다.
- 범용 화면보다 역할별 운영 화면을 우선한다.
- 모든 중요한 상태 전이는 로그로 남긴다.
- 공통 운영 레이어는 여러 도메인 기능 위에 재사용 가능해야 한다.
- 도메인 기능은 업종 특화 가치를 분명히 드러내야 한다.

## 구현 우선순위
1. Auth 연동과 보호된 콘솔
2. organization / membership / role / seat
3. 초대 기반 가입
4. 감사 로그와 관리자 콘솔
5. 도메인 기능별 사용자 귀속
6. 영업 claim, 승인, 파이프라인 집계
7. SSO 등 기업 확장 기능

## 결론
이 서비스는 `Supabase를 대체하는 범용 플랫폼`으로 가는 것이 아니라, `Supabase를 인프라로 쓰면서 그 위에 조직 운영과 설계공모 영업 도메인을 얹는 도메인 특화 B2B 운영 플랫폼`으로 가는 것이 맞다.
