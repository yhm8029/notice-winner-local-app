# Phase 1 운영 정책 분기표

- 문서 역할: 아카이브 노트
- 정본 여부: `archive`
- 이 문서가 답하는 질문: Phase 1 운영 인원 규모에 따라 어떤 운영 정책 분기를 고려해야 하는지
- 이 문서가 답하지 않는 질문: Phase 2 권한/조직 정책의 최종 기준, 제품 요구사항의 최종 정의
- 상위 기준 문서: [../../00_CANONICAL_INDEX_KR.md](../../00_CANONICAL_INDEX_KR.md)
- 충돌 시 우선 문서: [../../00_CANONICAL_INDEX_KR.md](../../00_CANONICAL_INDEX_KR.md)

## 목적
- Phase 1 웹 버전을 실제 몇 명이 사용할지에 따라 운영 정책을 분기한다.
- 여기서 운영 인원은 관리자 수가 아니라, 실제로 실행/취소/로그조회/결과다운로드를 할 사람 수를 의미한다.

## 공통 전제
- Phase 1은 현재 GUI 기능의 웹 이관이 목적이다.
- 로그인/권한 체계는 Phase 2에서 본격 도입한다.
- `requested_by`는 `not null` 유지한다.
- Phase 1 기본값은 `internal user` 고정 UUID 사용이다.

## 공통 원칙
- 외부 노출보다 내부 운영 안정성이 우선이다.
- 실행 이력은 누가 만들었는지 최소 수준이라도 추적 가능해야 한다.
- 취소는 단순 버튼 기능이 아니라 운영 정책이 필요한 행위다.

## 운영 인원 1명

### 적용 조건
- 실제 사용하는 사람이 1명
- 개인 내부 도구에 가까운 운영

### 권장 정책
- 로그인 없이 운영 가능
- `requested_by = internal user` 고정
- 보조 추적 필드 없이도 운영 가능
- 취소 버튼은 별도 제약 없이 허용 가능
- 접속 통제는 약하게 두어도 리스크 낮음

### 최소 운영 규칙
- 실행/취소/다운로드는 동일 사용자만 수행
- run ownership 문제를 별도로 처리하지 않음
- 로그와 summary만으로 운영 추적

### 장점
- 구현 가장 빠름
- 문서/스키마/프론트 단순

### 리스크
- 나중에 2명 이상으로 늘어나면 즉시 정책 보강 필요

## 운영 인원 2명 이상

### 적용 조건
- 실제로 웹을 사용할 사람이 2명 이상
- 공용 도구에 가까운 운영

### 권장 정책
- 로그인 없이도 시작할 수는 있으나, 운영 규칙은 필수
- `requested_by = internal user` 고정은 유지
- 추가 보조 추적값 필요
  - `operator_name`
  - 또는 `requested_by_label`
  - 또는 `meta_json.operator_name`
- 접속 통제 필요
  - Nginx IP allowlist
  - VPN 또는 내부망 한정

### 취소 정책
- 최소한 아래 중 하나를 정해야 함
  - 누구나 모든 run 취소 가능
  - 실행자만 취소 가능
  - 관리자 지정 인원만 취소 가능

### 권장 기본안
- 생성 시 `operator_name` 저장
- 취소도 `operator_name` 기록
- 운영 초기엔 “누구나 취소 가능”으로 두되 로그에 취소 주체 남김

### 장점
- 로그인 없이도 책임 추적 가능
- Phase 2 로그인 도입 전까지 버틸 수 있음

### 리스크
- 규칙 없이 운영하면 충돌 발생
  - A가 만든 run을 B가 취소
  - 누가 어떤 실행을 돌렸는지 불명확

## 정책 분기표

| 항목 | 운영 인원 1명 | 운영 인원 2명 이상 |
|---|---|---|
| 로그인 | 불필요 | Phase 1은 생략 가능하나 Phase 2 우선 검토 |
| requested_by | internal user 고정 | internal user 고정 |
| 보조 추적값 | 선택 | 필수 권장 |
| 취소 버튼 | 자유 허용 | 정책 정의 필요 |
| 접근 통제 | 선택 | 내부망/IP allowlist 권장 |
| 책임 추적 | 낮아도 운영 가능 | 최소 실행자 식별 필요 |

## Phase 1 권장 결론

### 1명 운영이면
- 가장 단순한 MVP로 간다
- 로그인 없음
- internal user 고정
- 취소/다운로드 자유

### 2명 이상 운영이면
- 로그인은 Phase 2로 미루더라도 아래는 Phase 1에 포함한다
  - `operator_name` 저장
  - 취소 주체 기록
  - 내부망 또는 allowlist 접근 제한

## 스키마 권장 보강

운영 인원 2명 이상 가능성이 있으면 아래 중 하나를 Phase 1부터 넣는 것을 권장한다.

### 옵션 A
- `pipeline_runs.operator_name text not null default ''`

### 옵션 B
- `pipeline_runs.meta_json`
  - `operator_name`
  - `client_ip`
  - `cancel_requested_by`

권장:
- 빠른 구현은 `operator_name` 단일 컬럼
- 유연성은 `meta_json`

## API 권장 보강

운영 인원 2명 이상이면 아래 필드를 권장한다.

### 실행 생성
- `operator_name`

### 취소 요청
- `requested_by_label` 또는 `operator_name`

### 로그
- 생성/취소 주체를 meta에 남김

## 최종 원칙
- “관리자 몇 명”보다 “실제로 실행 버튼 누를 사람 몇 명”이 먼저다.
- 운영 인원 1명이면 단순화가 맞다.
- 운영 인원 2명 이상이면 로그인 없이 가더라도 추적과 취소 정책은 반드시 넣어야 한다.

## 현재 결정
- Phase 1 운영 인원: 1명 (내부 단독 운영)
- 로그인: 적용하지 않음
- requested_by: internal user 고정 UUID 사용
- operator_name: 현재는 불필요
- 취소 버튼: 자유 허용
- 접근 통제: 내부 운영 기준, Nginx IP 제한은 선택
- 향후 확장: 다중 사용자 전환 시 operator_name 또는 동등한 보조 추적값, 취소 정책, 접근 제한을 추가 적용
