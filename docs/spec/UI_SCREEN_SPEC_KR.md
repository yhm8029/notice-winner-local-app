# UI 화면 명세서

- 문서 역할: UI 화면 명세서
- 정본 여부: `canonical`
- 이 문서가 답하는 질문: 어떤 화면과 패널이 실제로 존재하며, 각 영역은 무엇을 보여주고 어떤 상호작용을 허용하는가
- 이 문서가 답하지 않는 질문: API 필드 전체 정의, DB 컬럼 전체 정의, 내부 알고리즘 fallback 세부 순서
- 상위 기준 문서: [00_CANONICAL_INDEX_KR.md](../00_CANONICAL_INDEX_KR.md), [FUNCTIONAL_SPEC_KR.md](./FUNCTIONAL_SPEC_KR.md), [SYSTEM_DESIGN_KR.md](./SYSTEM_DESIGN_KR.md)
- 충돌 시 우선 문서: [00_CANONICAL_INDEX_KR.md](../00_CANONICAL_INDEX_KR.md)

작성일: 2026-03-22  
상태: 통합 초안 v1

## 1. 문서 목적

이 문서는 현재 웹 콘솔 구현을 재구축 가능한 수준으로 설명하는 화면 기준 문서다.

이 문서는 아래 두 문서의 내용을 정본 기준으로 재정리한다.

- [reference/operations/WEB_CONSOLE_PANEL_GUIDE_KR.txt](../reference/operations/WEB_CONSOLE_PANEL_GUIDE_KR.txt)
- [reference/operations/OPERATIONS_USER_CONVENIENCE_SPEC_KR.md](../reference/operations/OPERATIONS_USER_CONVENIENCE_SPEC_KR.md)

목표는 아래와 같다.

1. 재구축 담당자가 현재 웹 화면 구조를 코드 없이도 재현할 수 있어야 한다.
2. 어떤 패널이 사용자 모드에만 보이고, 어떤 패널이 관리자 모드에만 보이는지 명확해야 한다.
3. 버튼, 입력, 모달, 빈 상태, 다운로드 동작을 화면 기준으로 식별할 수 있어야 한다.

## 2. 화면 구성 원칙

1. 기본 진입점은 `/app/` 하나다.
2. 로그인 전과 로그인 후는 같은 경로에서 shell만 바뀐다.
3. 로그인 후에도 `실사용자 모드`와 `관리자 모드`가 공존하며, 모드에 따라 노출 패널이 달라진다.
4. 사용자에게는 업무에 필요한 최소 패널만 보여주고, 관리자에게는 운영/검증 패널을 추가로 노출한다.

## 3. 전역 UI 상태

### 3.1 인증 상태

- `unauthenticated`
- `authenticated but unauthorized`
- `authenticated and authorized`

노출 규칙:

1. `unauthenticated`이면 로그인 shell만 보인다.
2. `authenticated but unauthorized`이면 접근 차단 화면만 보인다.
3. `authenticated and authorized`이면 콘솔 shell이 보인다.

### 3.2 화면 모드

- `실사용자 모드`
- `관리자 모드`

노출 규칙:

1. 기본 진입은 실사용자 모드다.
2. 관리자 권한이 있는 경우에만 관리자 모드 전환 버튼을 사용한다.
3. 관리자 모드에서도 사용자 업무 패널은 유지할 수 있으나, 운영 패널을 추가 노출한다.

## 4. 최상위 화면 구조

로그인 후 화면은 아래 순서를 가진다.

1. 상단 hero/header
2. 운영 대시보드
3. 실행 생성 폼
4. 실행 상세
5. 최근 실행
6. 프로젝트 집계
7. 로그
8. 최신 리포트
9. 아티팩트/워크북
10. 트래커/영업 패널
11. 초대 및 사용자 관리

모든 패널이 항상 보이는 것은 아니며, 모드와 선택 상태에 따라 일부는 숨김/축약될 수 있다.

## 5. 로그인 Shell

### 5.1 목적

로그인 전 사용자를 보호된 콘솔 밖에 두고, 로그인/초기 운영자 등록/초대 수락 진입점을 제공한다.

### 5.2 표시 요소

1. 화면 제목
2. 설명 문구
3. 로그인 / 최초 운영자 등록 탭
4. 이메일 입력
5. 비밀번호 입력
6. 표시 이름 입력
   - 최초 운영자 등록 모드에서만 사용
7. 제출 버튼
8. 상태 메시지
9. 접근 차단 패널
10. 로그아웃 버튼
   - 차단 화면에서만 노출 가능

### 5.3 동작

1. 로그인 탭에서는 이메일 + 비밀번호로 로그인한다.
2. 최초 운영자 등록 탭에서는 bootstrap 이메일과 일치하는 경우에만 가입 가능하다.
3. 초대 토큰이 URL에 있으면 로그인 후 invitation accept 흐름을 이어서 처리한다.
4. 로그인 실패/권한 부족/초대 미존재는 모두 상태 메시지 또는 차단 패널로 보여준다.

## 6. 상단 Header / Meta 카드

### 6.1 공통 요소

1. 서비스 제목 `Project Tracker`
2. 서브 문구
3. 화면 모드 카드
4. 사용자 모드/관리자 모드 전환 버튼

### 6.2 관리자 모드에서만 보이는 요소

1. API 카드
2. 동기화 카드
3. Account 카드

### 6.3 사용자 모드에서 보이는 요소

1. 회원정보 버튼
2. 로그아웃 버튼

규칙:

1. 사용자 모드에서는 운영 디버그 성격의 `API`, `동기화`, `Account` 카드가 보이지 않는다.
2. 관리자 모드에서는 계정/역할/로그아웃 정보를 상단 메타 카드에서 본다.

## 7. 운영 대시보드

### 7.1 목적

운영자가 전체 상태를 빠르게 파악하는 요약 패널이다.

### 7.2 포함 요소

1. 실행 건수
2. 진행 중 건수
3. 실패 건수
4. tracker row 수
5. edited row 수
6. 최근 실패 실행
7. 실행 중인 리포트 작업

### 7.3 버튼

1. `대시보드 새로고침`

### 7.4 표시 규칙

1. 숫자 카드와 상태 카드 조합으로 구성한다.
2. 값이 없으면 empty state 문구를 보여준다.
3. 관리자 관점 요약 패널이므로 실사용자 모드에서는 중요도가 낮다.

## 8. 프로젝트 트랙커 실행 생성

### 8.1 목적

새 `project_tracker` 부모 실행을 만드는 시작 화면이다.

### 8.2 입력 필드

1. 시작일
2. 종료일
3. 계약일 힌트
4. 공고 번호
5. 공고명
6. 수요기관
7. 페이지당 행 수
8. 최대 페이지 수
9. API 범위
10. 고급 옵션
    - collect mode
    - 단계 지연
    - LLM 보정 여부
    - LLM 모델
    - LLM 최대 건수
    - export 병렬 워커

### 8.3 버튼

1. `실행 시작`
2. `초기화`

### 8.4 동작 규칙

1. 공고 번호, 공고명, 수요기관 중 하나 이상은 필수다.
2. `실행 시작`은 새 부모 run을 생성한다.
3. `초기화`는 폼만 리셋한다.

## 9. 실행 상세

### 9.1 목적

선택한 run의 현재 상태와 진행률, 요청값, 요약값, 에러값을 보여준다.

### 9.2 표시 요소

1. 실행 ID
2. 상태 배지
3. 진행률
4. run type
5. 단계 진행 바
6. 요청 JSON
7. 요약 JSON
8. 오류 JSON
9. 연결 실행 정보
10. tracker export 진행 카드
11. workbook 미리보기

### 9.3 버튼

1. `새로고침`
2. `실행 취소`
3. `수동 트래커 내보내기`
4. `트래커 엑셀 다운로드`

### 9.4 노출 규칙

1. 선택 run이 없으면 empty state를 보여준다.
2. 관리자 모드에서는 전체를 노출한다.
3. 실사용자 모드에서는 이 패널을 숨긴다.

## 10. 최근 실행

### 10.1 목적

과거 실행과 현재 실행을 목록으로 보고, 실행 상세 패널의 기준 run을 고르게 한다.

### 10.2 요소

1. 상태 필터
2. 실행 유형 필터
3. 날짜 범위 필터
4. 페이지 크기
5. 목록 새로고침
6. 페이지 이동
7. 실행 카드

### 10.3 동작 규칙

1. 카드 선택 시 실행 상세/로그/아티팩트/트래커 패널 기준이 함께 바뀐다.
2. `project_tracker`, `tracker_export`를 모두 지원한다.

## 11. 프로젝트 집계

### 11.1 목적

run 단위가 아니라 프로젝트 단위로 최근 결과를 다시 찾고, 실행 생성 폼에 조건을 재적용한다.

### 11.2 요소

1. 검색 입력
2. 새로고침 버튼
3. 프로젝트 카드
4. `조건 적용` 버튼

## 12. 로그 패널

### 12.1 목적

선택한 run의 단계별 메시지를 시간순으로 보여준다.

### 12.2 요소

1. 로그 새로고침
2. 로그 목록
3. 이벤트 상태

### 12.3 표시 규칙

1. 단계/레벨/메시지/생성시각이 함께 보여야 한다.
2. 진행 상황과 실패 원인을 모두 추적 가능해야 한다.

## 13. 최신 리포트 / 아티팩트

### 13.1 목적

동등성 테스트, GUI parity 검증, workbook 산출물 확인에 필요한 보조 영역이다.

### 13.2 요소

1. 최신 리포트 카드
2. report job 상태
3. artifact 목록
4. workbook preview
5. 다운로드 링크

### 13.3 규칙

1. 운영 필수라기보다 검증/보조 용도다.
2. 관리자 모드에서 우선적으로 사용한다.

## 14. 사용자 영업 홈

### 14.1 기본 구조

사용자 모드 핵심 홈은 아래 3개 영역으로 구성한다.

1. `내가 진행 중인 영업`
2. `회사 전체 진행 중인 영업`
3. `전체 영업 대상 프로젝트`

### 14.2 `내가 진행 중인 영업`

목적:

1. 현재 로그인한 사용자가 맡은 프로젝트를 직접 관리한다.

표시 요소:

1. 프로젝트명
2. 연면적
3. 공사비
4. 빌딩자동제어 추정금액
5. 설계사무소
6. 개찰예정일
7. 담당
8. 현장
9. 현재 상태 배지
10. 영업 시작일
11. 현재 담당 시작일
12. 영업현황 히스토리
13. 메모 입력 textarea
14. 이관 대상 선택

버튼:

1. `저장`
2. `계약 완료`
3. `영업 종료`
4. `해제`
5. `이관`
6. `엑셀 다운로드`

규칙:

1. 이 영역에서만 영업현황 수정 가능
2. 저장 시 새 메모가 append-only로 추가됨
3. 계약 완료 시 계약금액 입력 모달이 필수
4. 계약 완료/영업 종료/해제 후 목록 반영

### 14.3 `회사 전체 진행 중인 영업`

목적:

1. 현재 회사 전체가 잡고 있는 진행 중 프로젝트를 읽기 전용으로 본다.

표시 요소:

1. 사용자명/운영자
2. 프로젝트명
3. 주요 수치 요약
4. 최근 영업현황
5. 상태 배지
6. 엑셀 다운로드 버튼

규칙:

1. 전원 read-only
2. 본인이 수정한 메모 결과도 이 영역에 동기화되어 반영
3. 카드 높이는 `내가 진행 중인 영업` 카드와 시각적으로 맞춘다

### 14.4 `전체 영업 대상 프로젝트`

목적:

1. 아직 아무도 잡지 않은 신규 영업 대상만 보여준다.

표시 요소:

1. 프로젝트명
2. 연면적
3. 공사비
4. 빌딩자동제어 추정금액
5. 설계사무소
6. 개찰예정일
7. 담당
8. 현장

버튼:

1. `영업`
2. `연관 공고 열기`

규칙:

1. 진행 중/계약 완료/영업 종료 프로젝트는 여기서 숨긴다.
2. 영업현황 패널은 여기서 보이지 않는다.
3. 여기서는 `영업 시작`만 가능하고, 메모 입력은 불가하다.

## 15. 관리자 영업 패널

### 15.1 영업사원별 진행 현황

목적:

1. 어떤 사용자가 어떤 프로젝트를 맡고 있는지 집계한다.

표시 요소:

1. 사용자명
2. 진행 중 건수
3. 총 추정금액
4. 프로젝트 카드 목록
5. 강제 해제 버튼

규칙:

1. 자동 polling은 사용하지 않는다.
2. 영업 액션 발생 시 또는 수동 새로고침 시 갱신한다.
3. 진행 기간과 추정금액을 함께 보여준다.

### 15.2 종료/완료 정리

목적:

1. 계약 완료/영업 종료 결과를 별도 보관한다.

표시 요소:

1. 연도 그룹
2. 월 그룹
3. 계약 완료 섹션
4. 영업 종료 섹션
5. 계약금액 또는 추정금액

규칙:

1. 미래 연도는 노출하지 않는다.
2. 계약 완료 건은 계약금액을 콤마 형식으로 보여준다.
3. 영업 종료 건은 추정금액 기준으로 남긴다.

## 16. 사용자 초대 및 관리

### 16.1 목적

관리자가 사용자를 초대하고, 현재 회사 사용자를 관리한다.

### 16.2 초대 폼

입력 요소:

1. 이메일
2. 표시 이름
3. 역할
4. 팀명
5. 직책
6. 만료일(일)

버튼:

1. `초대 링크 생성`
2. `링크 복사`
3. `철회`

규칙:

1. 자기 자신 이메일로는 초대 생성 불가
2. 메일 자동 발송을 우선 시도
3. 실패 시 링크 fallback 허용
4. 목록에는 기본적으로 `pending`만 보여준다

### 16.3 사용자 목록

표시 요소:

1. 표시 이름
2. 이메일
3. 계정 활성/비활성
4. 소속 활성/비활성
5. 역할 선택
6. 소속 상태 선택
7. 팀명
8. 직책

버튼:

1. `저장`

규칙:

1. bootstrap/platform_admin 핵심 권한은 일반 관리자 화면에서 바꾸지 않는다.
2. 진행 중 영업이 남아 있으면 비활성화 전에 먼저 정리해야 한다.

## 17. 회원정보 모달

### 17.1 목적

현재 로그인 사용자의 계정 정보를 확인하고 수정한다.

### 17.2 표시 요소

1. 이메일
2. 회사
3. 역할
4. 상태
5. 표시 이름
6. 휴대폰
7. 회사 전화
8. 새 비밀번호
9. 현재 비밀번호

버튼:

1. `저장`
2. `닫기`

규칙:

1. 현재 비밀번호 없이는 저장되지 않는다.
2. 새 비밀번호는 길이 기준을 충족할 때만 변경된다.
3. 휴대폰/회사전화도 현재 비밀번호 확인을 요구한다.

## 18. 모달/대화상자 기준

### 18.1 계약금액 입력 모달

사용 시점:

1. `계약 완료` 버튼 클릭 시

규칙:

1. 숫자만 입력 허용
2. 입력 중 천 단위 콤마 자동 적용
3. 비어 있으면 저장 금지

### 18.2 일반 경고/에러 메시지

규칙:

1. 사용자 액션 성공/실패는 flash message 또는 패널 내 상태 메시지로 보여준다.
2. 초대 메일 실패는 fallback 가능성을 포함해 관리자에게 알려야 한다.

## 19. 자동 갱신 규칙

1. 대시보드, 실행 목록, 로그 등은 자동 새로고침을 가질 수 있다.
2. 영업사원별 진행 현황은 실시간 polling을 사용하지 않는다.
3. 영업 관련 집계는 영업 액션 발생 시에만 반영하거나 수동 새로고침한다.

## 20. 재구축 완료 기준

이 문서 기준으로 UI가 재구축되었다고 보려면 아래가 충족되어야 한다.

1. 로그인 전/후 shell 전환이 재현된다.
2. 사용자 모드와 관리자 모드의 패널 차이가 재현된다.
3. 사용자 영업 홈 3영역 구조가 재현된다.
4. 관리자의 초대/사용자 관리 화면이 재현된다.
5. 계약금액 모달, 회원정보 모달, 영업현황 append 흐름이 재현된다.
6. 종료/완료 정리의 연/월 그룹 구조가 재현된다.
7. `실행 상세`는 관리자 모드에서만 노출되는 현재 동작이 재현된다.
