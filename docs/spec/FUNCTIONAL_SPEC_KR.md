## 1. 문서 목적

이 문서는 현재 흩어져 있는 기능 요구사항을 하나의 정본 문서로 고정한다.

본 문서는 특히 아래 두 문서의 기능 관점을 묶는다.

- [reference/source/SAAS_FUNCTIONAL_SPEC_FROM_GUI_KR.md](../reference/source/SAAS_FUNCTIONAL_SPEC_FROM_GUI_KR.md)
- [reference/operations/PHASE2_SALES_CLAIM_AND_PIPELINE_SPEC_KR.md](../reference/operations/PHASE2_SALES_CLAIM_AND_PIPELINE_SPEC_KR.md) 중 사용자 행위 요구사항

## 2. 문서 경계

이 문서에 포함:

1. 사용자/관리자 기능
2. 화면에서 가능한 행동
3. 업무 흐름
4. 상태 변화에 따른 사용자 경험

이 문서에서 제외:

1. API payload
2. DB 스키마
3. 알고리즘 fallback 순서
4. 메일 SMTP 공급자 선택
5. 운영상 내부 감사 저장 방식

## 3. 사용자 역할

기능 관점에서 현재 고려하는 역할은 아래와 같다.

1. `platform_admin`
2. `org_admin`
3. `org_member`

기능 차등의 기본 원칙:

1. `platform_admin`은 전역 운영 기능에 접근한다.
2. `org_admin`은 회사 사용자/초대/조직 운영 기능에 접근한다.
3. `org_member`는 실무 기능을 사용한다.

## 4. 핵심 기능 묶음

### 4.1 파이프라인 실행

사용자는 아래를 할 수 있어야 한다.

1. 실행 요청 생성
2. 실행 상태 확인
3. 실행 취소
4. 로그 확인
5. 산출물 다운로드

### 4.2 트래커 결과 확인

사용자는 아래를 할 수 있어야 한다.

1. 트래커 결과 보드 확인
2. 프로젝트 현황 탐색
3. 관련 공고 열기
4. 필요한 경우 인라인 수정 또는 관리자 수정

### 4.3 영업 파이프라인

사용자는 아래를 할 수 있어야 한다.

1. 영업 대상 프로젝트 목록 확인
2. 특정 프로젝트에 대해 영업 시작
3. 본인 담당 프로젝트의 영업현황 메모 추가
4. 본인 담당 프로젝트 해제
5. 계약 완료 또는 영업 종료 처리
6. 필요 시 다른 사용자에게 이관

### 4.4 회사/관리 기능

관리자는 아래를 할 수 있어야 한다.

1. 사용자 초대
2. 사용자 역할/상태 변경
3. 영업 현황 집계 확인
4. 강제 해제/강제 이관
5. 회원정보 및 회사 사용자 목록 관리

### 4.5 인증/회원정보

사용자는 아래를 할 수 있어야 한다.

1. 로그인
2. 로그아웃
3. 회원정보 확인
4. 회원정보 수정
5. 비밀번호 변경

### 4.6 초대 기반 온보딩

관리자는 아래를 할 수 있어야 한다.

1. 사용자 초대 생성
2. 초대 링크 확인 또는 전달
3. 초대 상태 확인
4. 필요 시 초대 철회

초대받은 사용자는 아래를 할 수 있어야 한다.

1. 초대 링크로 진입
2. 초대 이메일과 같은 계정으로 로그인 또는 가입
3. 초대 수락 후 회사 사용자로 활성화

## 5. 주요 화면 기준

### 5.1 사용자 모드

사용자 모드는 아래 중심으로 구성한다.

1. 내가 진행 중인 영업
2. 회사 전체 진행 중인 영업
3. 전체 영업 대상 프로젝트

### 5.2 관리자 모드

관리자 모드는 아래 기능을 포함한다.

1. 실행 상세
2. 초대 및 사용자 관리
3. 영업사원별 진행 현황
4. 종료/완료 정리

### 5.3 사용자 홈 기본 레이아웃

사용자 홈은 아래 우선순위로 정보를 배치한다.

1. 내가 진행 중인 영업
2. 회사 전체 진행 중인 영업
3. 전체 영업 대상 프로젝트

배치 원칙:

1. `내가 진행 중인 영업`과 `회사 전체 진행 중인 영업`은 같은 화면에서 동시에 본다.
2. `내가 진행 중인 영업`에서만 영업현황 메모를 입력/수정할 수 있다.
3. `회사 전체 진행 중인 영업`은 읽기 전용으로 보며, 상태 공유용 뷰로 사용한다.
4. `전체 영업 대상 프로젝트`는 신규 영업 대상만 보여주며 `영업 시작` 행동만 허용한다.

### 5.4 사용자 관리 화면 기준

관리자 화면의 사용자 관리 영역은 아래 기능을 포함한다.

1. 사용자 초대
2. 초대 목록 조회
3. 역할 변경
4. 소속 상태 변경
5. 팀명/직책 수정
6. 비활성화 전 진행 영업 보유 여부 확인
7. 플랜별 초대/가입 한도 요약 확인

초대 기능은 초기에는 `메일 발송 + 링크 fallback`을 함께 지원하고, 추후 메일 공급자 전환을 허용한다.

플랜 한도 기준:

1. 조직은 `A/B/C` 플랜 중 하나를 가지며, 현재 기준 예시는 `A=5`, `B=10`, `C=100`이다.
2. `pending` 초대 수는 초대 한도에 포함한다.
3. `active` 사용자 수가 가입 한도에 도달하면 새 초대를 만들 수 없다.
4. 관리자 화면은 `현재 가입 수 / 한도 / 남은 수`와 `현재 pending 초대 수 / 한도 / 남은 수`를 함께 보여준다.
5. 한도 초과 시 상위 플랜 업그레이드 안내 문구를 보여준다.

### 5.5 사용자 계정/보안 화면 기준

회원정보 화면은 아래를 포함한다.

1. 이메일, 회사, 역할, 상태 조회
2. 표시 이름 수정
3. 휴대폰/회사전화 수정
4. 비밀번호 변경

보안 경험 규칙:

1. 회원정보 변경은 현재 비밀번호 확인을 요구한다.
2. 사용자 모드에서도 로그아웃은 항상 가능해야 한다.
3. 공유된 기기에서 열린 세션을 고려해, 민감 변경은 재확인을 거친다.

## 6. 주요 업무 규칙

1. 하나의 프로젝트는 동시에 한 명만 영업 담당 가능하다.
2. 일반 사용자는 본인 담당 프로젝트만 수정 가능하다.
3. 메모는 삭제보다 정정 추가를 기본으로 한다.
4. 영업 owner 변경은 release보다 transfer를 중심으로 본다.
5. 계약 완료 시 계약금액 입력이 필요하다.
6. 전체 영업 대상 프로젝트에는 이미 진행 중/종료/완료된 건을 노출하지 않는다.

### 6.1 역할별 기능 범위

`platform_admin`

1. 전역 운영/점검 기능 접근
2. 회사 및 사용자 구조 점검
3. 관리자 기능 전부 접근 가능
4. `org_admin`, `org_member` 초대 가능
5. `org_admin`, `org_member` 초대 목록/링크/초기 암호/철회까지 관리 가능

`org_admin`

1. 초대/사용자 관리 가능
2. 영업 집계 확인 가능
3. 강제 해제/강제 이관 가능
4. 종료/완료 정리 확인 가능
5. `org_member`만 초대 가능
6. `org_admin` 대상 초대는 조회/철회할 수 없다

`org_member`

1. 자기 영업 claim 가능
2. 본인 담당 영업 메모 추가 가능
3. 본인 담당 영업 해제/이관 요청 가능
4. 회사 전체 진행 상황은 읽기 전용

### 6.2 핵심 상태 변화

영업 파이프라인 기능은 아래 상태 변화를 사용자에게 일관되게 보여야 한다.

1. 미배정
2. 진행 중
3. 계약 완료
4. 영업 종료

초대/계정 기능은 아래 상태 변화를 사용자에게 일관되게 보여야 한다.

1. 초대 대기 중
2. 초대 수락 완료
3. 초대 만료
4. 초대 철회
5. 계정 활성
6. 계정 비활성

사용자 경험 규칙:

1. 미배정 상태에서만 신규 영업 시작 가능
2. 진행 중 상태는 담당자 1인에게 잠김
3. 계약 완료/영업 종료는 대상 목록에서 제외
4. 종료/완료 내역은 별도 정리 영역에서 연도/월 기준으로 확인

## 7. 완료 기준

이 문서 기준으로 완료되려면 아래가 충족되어야 한다.

1. 사용자가 로그인 후 자기 역할에 맞는 화면을 본다.
2. 영업 파이프라인의 시작/메모/이관/완료/종료 흐름이 화면에서 일관되게 동작한다.
3. 관리자는 초대/사용자 관리/영업 집계를 수행할 수 있다.
4. 기능 차이가 화면과 권한에 일관되게 반영된다.
5. 로그인/초대/회원정보/비밀번호 변경 흐름이 사용자 입장에서 독립적으로 완결된다.
6. 재구축 시 본 문서만으로 필요한 화면군과 사용자 행위를 식별할 수 있다.
7. 사용자 모드/관리자 모드의 화면 차이가 주요 업무 기준으로 설명 가능해야 한다.
