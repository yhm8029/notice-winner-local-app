# Phase 2 논의 요약 및 현재 설계 핸드오프

## 문서 목적
- 이 문서는 지금까지 사용자와 합의한 Phase 2 방향, 사업적 의도, 운영 정책, 현재 설계 상태를 다른 AI나 다른 구현 세션에 그대로 넘기기 위한 핸드오프 문서다.
- 세부 구현 문서가 아니라 `왜 이렇게 가는지`, `무엇을 이미 결정했는지`, `무엇이 아직 남았는지`를 빠르게 파악할 수 있도록 정리한다.

## 현재 서비스에 대한 공통 이해
- 이 서비스는 단순한 프로젝트 트래커가 아니라, 설계공모/영업/프로젝트 추적 도메인에 특화된 B2B 운영 플랫폼으로 간다.
- 고객이 돈을 내는 이유는 범용 인증이나 범용 DB가 아니라, `공고 수집 -> 당선사/계약 정보 정리 -> tracker 생성 -> 영업 운영 -> 관리자 집계`가 하나의 워크플로로 돌아가기 때문이다.
- 따라서 Supabase를 대체하는 범용 BaaS를 만드는 것이 목표가 아니라, Supabase 위에 조직 운영과 도메인 기능을 얹는 것이 목표다.

## Phase 2의 큰 목표
- 로그인/세션을 붙인다.
- 여러 회사가 안전하게 같은 서비스를 쓰게 만든다.
- 회사 단위 고객 운영이 가능하게 만든다.
- 누가 어떤 run, tracker 수정, 영업 활동을 했는지 추적 가능하게 만든다.
- 향후 회사 SSO까지 확장 가능한 구조를 만든다.

즉 Phase 2는 `로그인 추가`가 아니라 `다중 조직 B2B SaaS 운영 구조`다.

## 사업/운영 정책 방향

### 판매 단위
- 고객은 `organization` 단위로 관리한다.
- 과금과 운영의 기준도 `organization`이다.

### 계정 정책
- 기본 원칙은 `1인 1계정`이다.
- 공용 계정 돌려쓰기를 어렵게 만들어야 한다.
- 계정은 자유가입이 아니라 `초대 기반 가입`으로 간다.
- 초대된 이메일과 실제 로그인에 쓰는 이메일은 동일해야 한다.

### 로그인 수단
- 기본 인증 플랫폼은 `Supabase Auth`다.
- 로그인 수단은 `Google 로그인 + 이메일 로그인`을 같이 지원하는 방향이다.
- 단, 둘 다 아무 이메일이나 허용하는 것이 아니라 `초대된 이메일`과 일치할 때만 허용한다.
- 같은 검증된 이메일이면 이메일 로그인과 Google 로그인은 동일 사용자로 연결되는 방향으로 설계한다.

### 세션 정책
- 현재 구현은 signed cookie + refresh 기반 세션 유지다.
- `단일 활성 세션(single active session)`은 운영 요구가 더 명확해질 때 후속 hardening으로 분리한다.
- 즉 Phase 2 안정화 범위에서는 새 로그인 시 기존 세션 강제 종료를 구현하지 않는다.

### 좌석 정책
- 회사별 `active_user_limit`, `pending_invite_limit`를 둔다.
- 활성 사용자 수 또는 pending 초대 수가 한도를 초과하면 추가 활성화/초대를 차단한다.
- 이 정책은 B2B 과금과 운영 통제를 위한 핵심 장치다.

### 향후 기업 확장
- 큰 고객사를 위해 `회사 SSO(Google Workspace / Microsoft Entra / Okta 등)` 확장 경로를 열어둔다.

## 현재까지 사용자와 합의한 UX 원칙

### 로그인 UX
- 최종 메인페이지를 먼저 만드는 것보다 `로그인 성공 여부를 즉시 확인할 최소 auth shell`을 우선한다.
- 로그인 전에는 로그인 화면만 보이고, 로그인 후에는 기존 콘솔로 들어간다.
- 로그인 성공 시 상단에서 최소한 `이메일 / 역할 / 회사 / 로그아웃`이 보이게 한다.

### 모드 노출
- 사용자 모드에선 운영용 카드가 보이면 안 된다.
- API, 동기화, ACCOUNT 같은 운영자성 정보는 관리자 모드에서만 보이게 한다.

### tracker 보드 편집
- 사용자 모드의 tracker 결과 보드에서 필드를 직접 수정할 수 있어야 한다.
- 셀 편집은 인라인으로, 저장 시 감사 로그가 남아야 한다.
- 이 흐름은 이후 로그인 사용자 ID 기준으로 기록되어야 한다.

## 영업 claim 기능에 대한 합의사항

### 기능 목표
- 여러 영업 사용자가 같은 프로젝트를 동시에 잡지 못하게 한다.
- 누가 어떤 프로젝트를 담당하는지 모두가 볼 수 있어야 한다.
- 담당자는 `영업현황`을 직접 입력할 수 있어야 한다.
- 관리자는 영업사원별 진행 건과 총 추정 금액을 집계해서 볼 수 있어야 한다.

### 잠금 기준
- 잠금 기준은 `entry_id`가 아니라 `project_id`다.
- 이유는 같은 프로젝트가 보드에서 여러 row로 보일 수 있기 때문이다.
- 영업 충돌을 막는 기준은 row가 아니라 프로젝트다.

### 해제 정책
- 본인은 자기 claim을 해제할 수 있다.
- 관리자는 강제 해제할 수 있다.

### 영업현황 위치
- 사용자 카드에서 `담당`, `현장` 아래 줄에 `영업현황`을 둔다.
- owner만 수정 가능하다.

### 기록 항목
- claim 시각
- owner
- sales_note(영업현황)
- 추정 금액
- release 여부
- claim / note_update / release / force_release 이벤트 로그

## 최소 영업 액션 모델 확장
현재 `sales claim` 설계 위에 아래 최소 액션 필드를 얹는 방향으로 확장한다.

1. `sales_priority`
2. `next_action`
3. `due_date`
4. `sales_status`

정리:
1. `owner`는 기존 claim owner를 사용한다.
2. `memo`는 기존 `sales_note`를 재사용한다.
3. `sales_status`는 `claimed_at / released_at`와 다른 개념이다.
4. 상세 설계는 [PHASE2_SALES_ACTION_MINIMUM_MODEL_KR.md](./PHASE2_SALES_ACTION_MINIMUM_MODEL_KR.md)를 따른다.

## 영업 claim 저장소 방향

### 1차
- 빠른 UX 검증을 위해 in-memory로 먼저 구현했다.

### 현재 방향
- 최종적으로는 Supabase/Postgres 영속 저장으로 간다.
- 관련 테이블:
  - `project_sales_claims`
  - `project_sales_claim_events`

### 현재 검증 상태
- Supabase 테이블 생성 후, 실제 원격 DB 기준으로 `claim -> note_update -> release` 저장이 검증되었다.
- bootstrap 사용자 `yhm8029@gmail.com` 기준으로 검증했다.
- 원격 테이블에서 다음 항목이 확인되었다.
  - owner_email
  - sales_note
  - claimed_at
  - released_at
  - 이벤트 로그 3건(`claim`, `note_update`, `release`)

즉 현재 sales claim은 더 이상 in-memory 가정이 아니라 Supabase 영속 저장으로 전환 가능한 상태다.

## 다른 AI에게 전달해야 할 핵심 요약
- 이 서비스는 범용 인프라가 아니라 도메인 특화 B2B 운영 플랫폼을 지향한다.
- Supabase는 인프라 레이어로 쓰고, 조직 운영/권한/도메인 규칙은 우리 코드가 가진다.
- Phase 2의 핵심은 로그인 기능 추가가 아니라 조직/역할/초대/seat/감사 로그 구조다.
- sales claim은 `project_id` 기준 잠금, 본인 해제/관리자 강제 해제, 영업현황 메모, 관리자 집계를 전제로 설계되었다.
- 사용자와 이미 합의된 운영 정책은 `1인 1계정`, `초대 기반`, `초대 이메일 일치`, `active_user_limit / pending_invite_limit`, `향후 SSO`, `단일 활성 세션은 후속 검토`다.
